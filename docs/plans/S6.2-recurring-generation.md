# S6.2 — Generation engine & projection hook

> Story: [Notion S6.2](https://app.notion.com/p/37ca7227e26f8125ac86dd955382f0b7) · Design: [F6 Recurring](https://app.notion.com/p/37ca7227e26f810b86b2e9b846dfc861) · Design system: [canonical](https://app.notion.com/p/37da7227e26f81f28e85fa5c6d1d38f8) · PR: #16 (base `main`)

> **Final story of the billd build.** Closes F6 and the application.

## Objective

Generate real expenses from recurring templates via an idempotent catch-up-on-access routine, and fold scheduled-but-unposted recurring amounts into the dashboard projection.

## Data Model

No schema change — `RecurringTemplate` + `PostingLedger` exist (S6.1). `Expense` gains a nullable `generatedFromId` so generated rows are traceable (small additive migration).

## Approach

1. Prisma: add `Expense.generatedFromId String?` (+ index); migration.
2. **`src/lib/recurringGeneration.ts`** — the engine (pure planner + DB applier split for testing):
   - `dueOccurrences(template, from, to)` — pure: every scheduled date in `(from, to]` for an active template, walking `nextOccurrence`. Cap at 12 (the catch-up cap).
   - `generateDue(userId, now)` — for each active template, find occurrences since the last ledger entry (or startDate) up to today; for each, **insert a PostingLedger row + an Expense in one transaction**; the `@@unique(templateId, scheduledDate)` constraint makes it idempotent (catch a unique violation → skip, already posted). Returns count generated + whether the 12-cap was hit.
3. **Catch-up on access** — call `generateDue` from the dashboard page load (server) before computing the summary, so visiting the app posts due expenses. Cheap: one indexed query per template; no-op when nothing is due.
4. **Projection hook** — extend the dashboard projection: add scheduled-but-unposted occurrences for the rest of the current month to the projected figure (uses `dueOccurrences` from today→month-end). Wire into `projectionStatus` via an added `scheduledRemainingCents` argument (kept backward-compatible / unit-tested).
5. **12-cap confirm** — if a template has >12 unposted occurrences (long absence), post the most recent 12 and surface a one-time notice on the recurring page ("Some past occurrences were skipped"). (Design's interactive confirm simplified to a capped post + notice for V1 — non-blocking; logged as a deviation.)

## Generation contract (Design F6 W2)

| Rule | Spec |
|---|---|
| Idempotency | unique `(templateId, scheduledDate)`; re-runs/concurrent loads can't double-post |
| Paused | `active=false` → no occurrences accrue while paused |
| Deleted generated expense | ledger row remains → not regenerated |
| Catch-up cap | ≤12 postings per template per run; older skipped with a notice |
| Projection | dashboard projection adds scheduled-but-unposted current-month occurrences |
